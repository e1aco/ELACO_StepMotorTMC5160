# 工作流: new 轻档（quick）

> 适用：< 2h 工作量、单文件改动、bugfix、参数调优、小特性
> 产出：1 个 `quick-plan.md`，立即可执行

## 核心理念
**最小可执行计划**：不发散、不头脑风暴、只确认目标和边界 → 直接 verify。

**追问原则**（轻量版）：
- 能通过查代码/文件确认的事实，自己查，不问用户
- 有歧义时逐个澄清，不一次问多个问题

## 流程

```
1. AI 输出 quick-plan 草稿（基于功能描述）
   ↓
2. 用户「继续」或修改
   ↓
3. 保存 quick-plan.md
   ↓
4. 更新 state.md / project-spec.md
   ↓
5. 提示 /em verify s<N> 进入验证
```

## 步骤 1: AI 输出 quick-plan 草稿

**前置检查 — drv 文件决策**（嵌入式项目时）：
- 判断需求是否涉及引脚/外设配置（见 `ela_rules_quick.md` 决策流程）
- 涉及硬件 → 快速输出文件结构：`ela_xxx_drv.c/h`（HAL）+ `ela_xxx_usr.c/h`（业务）
- 纯计算 → 单文件 `ela_xxx.c/h`

**模板**:

```markdown
# quick-plan: S<N>-<slug>

## 一句话目标
<10-30 字以内，描述完成后世界变成什么样>

## 改动清单（≤ 5 项）
- [ ] 文件1 - 改动摘要
- [ ] 文件2 - 改动摘要

## 验证方式
<如何确认完成了：跑某测试 / 启动看现象 / 用户口述>

## 不做的事（边界）
- 不重构无关代码
- 不引入新依赖
```

输出后提示：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 输入 `继续` 采用此计划
✏️  或直接修改任意字段
🔄 输入 `升档 中` / `升档 重` 切换档位
❌ 输入 `取消` 退出
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 步骤 2: 保存

- 路径：`<STATE_DIR>/discussion/<YYYYMMDD>-<slug>/quick-plan.md`
- **不**生成 status.json（轻档无阶段）

## 步骤 3: 同步状态

- `state.md` 更新：
  - 当前步骤 = S<N>（开发中）
  - 下一步动作 = `/em verify s<N>`
- `project-spec.md` 步骤表追加一行：`| S<N> | <slug> | 🚧 开发中 | YYYY-MM-DD |`

## 步骤 4: 收尾输出

```
🎉 S<N> 计划完成（轻档）

📄 计划文件: <STATE_DIR>/discussion/<...>/quick-plan.md
📝 改动清单: <N> 项
✅ 验证方式: <一句话>

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
下一步:
  开始编码 → 完成后 /em verify s<N>
  🔧 Keil 项目注意：编码时新增 .c 文件后，自动执行 keil-config 同步到工程
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 边界 / 升档触发

如在轻档中发现：
- 改动 > 5 个文件
- 跨多个子系统
- 涉及硬件外设新接入

→ AI 主动提示：「此需求可能超出轻档范围，建议升档为中档（`/em new ... --std`）」
